### 2026-06-16 画像が表示されなくなった不具合を修正（本番での画像の配信方法を変更）

- **症状**: 「画像生成されなくなった」。調べると、AIの画像生成じたいは成功していて、画像ファイルもちゃんと保存されていた（19件すべて生成済み）。なのに新しく作った画像だけ画面に出ない状態だった。
- **本当の原因**: 本番モード（`next start`）の Next.js 16 は、画像置き場の `public/` フォルダを「**ビルドした時点の中身**」で配信する。そのため、ビルドより**後**にAIが作って保存した画像は、ファイルはディスクにあるのに配信されず404（見つからない）になっていた。開発モード（`next dev`）では毎回ディスクから配信されるので、この問題に気づきにくかった。「前は出てたのに」は、最後のビルド時に存在した画像は出る／その後に作った画像だけ出ない、という切り分けで確定。
- **やったこと**: 画像を `public/` の静的配信に頼るのをやめ、**リクエストのたびにディスクから読んで返す専用の入口**（`/illustrations/(画像id)`）を新設。これならビルドの前後に関係なく、いつ作った画像でも表示できる。
  - 既存19件の画像URLも新しい入口に付け替え（404のままだった分も復活）。
- **どこを直したか**: `app/illustrations/[id]/route.ts`（新規・画像を配信するルートハンドラ）／`lib/images.ts`（保存後に返すURLを `/generated/xxx.png` → `/illustrations/xxx` に変更）／`.data/terms.json`（既存の画像URLを移行。バックアップは `scratch/terms.backup.json`）。画面側（用語ページ・カード）は `imageUrl` をそのまま使うだけなので変更なし。
- **確認したこと**: 型チェック エラー0／テスト47件パス／本番ビルド成功。**決め手の検証**: ビルド後にダミー画像を置いて2経路で取得し、旧・静的パスは404・新・ルートは200になることを確認（=ランタイム生成画像が確実に出る）。さっきまで404だった「Cicero型」の画像も新URLで200配信を確認。不正なid（`../` 等）は404で弾くこともチェック済み。
- **メモ**: 画像の保存先が将来ストレージ（Supabase 等）に変わっても、直すのは `lib/images.ts` とこのルートハンドラだけで済む。

### 2026-06-16 議事録の用語抽出を「候補を出す→選んで登録」に変更

- **きっかけ**: 「議事録の中からAIが単語を抽出して候補として挙げてほしい」という要望。これまでは抽出と同時に**全部まとめて自動登録**していた（選べない・要らない用語まで登録・AI料金もかさむ）。
- **やったこと**: 2段階に分けた。
  1. **用語の候補を出す**：文章からAIが専門用語の候補を挙げるだけ（登録はしない／安いAI1回）。すでに辞書にある語は「登録済み」として表示し、その用語ページへ開ける。
  2. **選んで登録**：候補をチェックで選び、選んだ分だけ説明・関連ワードを作って登録（未登録のものは既定でチェック済み）。
- **効果**: 要らない用語を登録しない＝**料金と手間を節約**、ユーザーが中身を制御できる。
- **どこを直したか**: `app/actions.ts`（`extractAndCreateTermsAction` を廃止し、`suggestTermsFromTextAction`＝候補抽出のみ／`createSelectedTermsAction`＝選択分だけ登録に分割）／`components/TermExtractor.tsx`（新規・候補選択UI）／`app/minutes/[id]/page.tsx`・`app/terms/extract/page.tsx`（差し替え）／旧 `components/ExtractForm.tsx` は撤去／`components/SubmitButton.tsx`（`disabled` 対応）。
- **確認したこと**: 型チェック エラー0／テスト47件パス／本番ビルド成功／dev 実機で `/terms/extract` の新UI（候補を出すボタン・旧自動登録ボタンが無いこと）を確認。AIの候補抽出・登録の実行はOpenAI課金を伴うため未実行（ロジック・ビルドは確認済み）。

- **どこを直したか**: 既定値は `lib/config.ts`（`TEXT_MODEL` / `IMAGE_QUALITY` / `TRANSCRIBE_MODEL`）。`.env.local` で上書き可能（`OPENAI_TEXT_MODEL` / `OPENAI_IMAGE_QUALITY` / `OPENAI_TRANSCRIBE_MODEL`）。
- **注意**: もし `.env.local` に `OPENAI_TEXT_MODEL=gpt-5.4-mini` 等が書いてあると、そちらが優先される。安くするなら該当行を消すか nano に直す。
- **確認したこと**: 型チェック エラー0。

### 2026-06-13 見た目をプロ級に刷新＋機能を4つ追加

- **やったこと（見た目）**:
  - アプリ全体の見た目を「クリーン・ミニマル」（Notion / Linear のような、余白を活かした落ち着いた雰囲気）に作り替えた。色・余白・角丸などの“決まりごと”を1か所（`app/globals.css`）にまとめ、各画面はその決まりだけを使うようにした。配色を変えたくなっても、基本この1ファイルを直すだけで全体に反映される。
  - よく使う部品（ボタン・カード・入力欄・タグ・読み込み中の仮表示など）を `components/ui/` に共通化した。
- **やったこと（新機能4つ）**:
  1. **ダークモード**: 画面右上の月/太陽ボタンで切替。選んだ状態は記憶され、次に開いたときも最初から正しい明るさで出る（パッと切り替わる際のチラつきなし）。
  2. **タグ分類＋絞り込み**: 用語に「分野タグ」（例：AI / 統計 / Web）が付く。AIが登録時に自動で付け、編集でも直せる。一覧の上のタグを押すと、その分野だけに絞り込める。
  3. **関連ワードの相互リンク**: 関連ワードがすでに辞書にある用語なら、その用語ページへ直接ジャンプできる。さらに詳細ページに「この用語とつながる用語」（自分を関連ワードに挙げている用語）も表示し、辞書が“つながり”として育つようにした。
  4. **議事録から用語を自動登録**: ヘッダーの「議事録から登録」から、会議メモなどを貼り付けると、AIが専門用語を見つけて、まとめて辞書に登録する（1回で最大12件）。
- **使い心地の工夫**:
  - 用語を登録してAIが考えている数十秒の間、完成後の見た目を“うすい仮表示（スケルトン）”で先に見せ、いま何をしているか（意味を調べる→関連ワード→説明→イラスト）を順に表示。待ち時間が短く感じられ、安心できる。
  - からっぽの画面（用語ゼロ・検索0件）も、次にやることが分かる案内に作り込んだ。
- **決めたこと・理由**:
  - 見た目の方向は「クリーン・ミニマル」、アクセント色はインディゴ（青紫）に決定（リサーチして比較した結果。詳細は `docs/research/2026-06-13_UI-UXモダン化リサーチ.md`）。
  - 議事録の一括登録は、料金を抑えるため1回の件数に上限を設け、重いイラスト生成はその場では行わない（各用語のページから後で作れる）。録音→文字起こしは今回は入れず、まずは文章貼り付けから。
- **確認したこと**: 型チェック（エラー0）・テスト（16件パス）・本番ビルド成功。開発サーバーを起動し、未ログインは `/login` へ自動で飛び、ログイン後はトップ・新規登録・議事録ページが正しく表示され、ダークモードの切替も効くことを確認した。
- **暫定対応（前回から継続）**: データ保存はローカルファイル、ログインは簡易版のまま（DB・本認証が決まったら差し替える）。
- **次にやること**: データベース（保存先）と本認証を決める。タグの自動付与や議事録抽出の精度を実データで確かめる。
