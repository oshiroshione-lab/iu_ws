IT用語集
用語名を入れるだけで、AIが「辞書の1ページ」を作る

インターン3人チームの社内辞書。覚えたいIT用語を入力すると、AIが比喩を使わない説明文・図解イラスト・関連ワード・分野タグを自動で作り、3人で共有・確認・コメントしながら育てる。議事録づくり（録音→文字起こし→用語抽出）も同じアプリの中でできる。この資料は、作者自身がこのアプリの仕組みを正しく理解し、人に説明できるようになるためのもの。

56 枚生成済みイラスト（手元データ）

1これは何か（一番の主題）

「用語名を入れるだけで、比喩を使わないわかりやすい1ページ（説明・図解・関連語・タグ）がAIで自動的にできあがる」——これがこのアプリの核。3人の小さなチームが、学んだIT用語を出し合って育てる“私たちの辞書”。

誰のための、何のアプリか

iU のインターン3人チームが、新しく学んだ専門用語を共有・保存するための社内ツール。一般公開や外部登録は作らない（内輪利用が前提）。PCのブラウザで使う。AIの役割は「調べる手間をなくし、初心者にも分かる説明をそろえる」こと。

2つの柱

📖

用語辞典

用語名を入力 → AIが説明・イラスト・関連ワード・タグを自動生成。一覧・検索・編集・いいね・コメント・確認済みバッジで、3人で育てる。

🎙️

議事録

ブラウザのマイクでリアルタイム文字起こし → AIで要約・決定事項・ToDoに整える → 本文から専門用語を抜き出して辞書に登録。

中核ルール（絶対）: AIが作る説明文には比喩・たとえ話を使わない。さらに「キャッシュ」のように他業界でも使う言葉でも、必ずIT・ソフトウェア分野の意味で説明する。これは仕様の中心で、AIへの指示文（lib/ai.ts）に必ず入っている。
失敗を握りつぶさない: イラスト生成に失敗したら、テキストだけ表示して「画像を生成できませんでした（後でやり直せます）」と出す。関連ワードが0件でもエラーにせず「見つかりませんでした」と表示する。

3用語を1件登録すると、裏で何が起きるか

登録ボタンを押すと、まず「説明文」までを作ってすぐ用語ページへ進む。時間のかかるイラストは“裏で”作り、完成したら画面の右下に通知する。だから待たされない。処理の入口はすべて app/actions.ts の Server Action。

裏付け（コードと一致）: app/actions.ts の createTermAction が、重複チェック → researchTerm（説明・関連語）→ chooseTag（タグ）→ 保存 → redirect の順に動き、イラストは after(...) の中で generateIllustrationBase64 → saveGeneratedImage を実行している。

イラストは“雰囲気の絵”ではなく“図解”

画像の役割は「その用語を絵で説明する」こと。そこで二段構えにしている。まず安いテキストAIに「概念を図解で説明する絵の設計（英語）」を書かせ、その設計を画像AIに渡す。画像AIは文字を崩しやすいので、長い文章は入れず、矢印・アイコン・箱（入力→処理→出力 など）で意味を伝えるよう指定している。

4議事録のしくみ（録音→整える→用語抽出）

もう一つの柱。マイクで話すとその場で文字が出る（リアルタイム文字起こし）。本文ができたらAIで要約・決定事項・ToDoに整え、さらに本文から専門用語を抜き出して辞書に登録できる。

料金の配慮: 用語抽出は2段階。まず「候補を挙げるだけ」（安いAI1回）→ 画面で選んだ分だけ説明を作って登録。要らない用語にAI費用を使わない。リアルタイム文字起こしは便利だが費用はやや高め、という点はメモしている。

5アーキテクチャ構成図（どこで何が動くか）

中心は1つ＝Cloud Run 上の Next.js サーバー。ブラウザはこことやり取りし、サーバーが外部（OpenAI・PostgreSQL・Cloud Storage）をまとめて呼ぶ。AIの鍵・DBの鍵はサーバー側だけに置く。

要件定義書との違い（正直に）: 当初は保存先・認証に Supabase、公開先に Vercel を「提案」していた。実際の今は PostgreSQL ＋ Google Cloud（Cloud Run / Cloud Storage） で動いていて、ログインは廃止済み。要件定義書のこの部分は提案段階の記述として残っている。

6差し替えできる設計（保存・画像・認証・AI）

「あとで決める／あとで替える」ものは、すべて1か所に集めてある。だから手元では設定なしのまま動かせ、本番では環境変数を足すだけで、共有DB・クラウド保存に切り替わる。画面側のコードは変えなくていい。

役割集約している場所既定（手元・設定なし）本番への切替（環境変数）
データ保存lib/store/ローカルJSON（.data/*.json）DATABASE_URL／PG* があれば PostgreSQL
イラスト画像lib/images.tspublic/generated/ に保存ASSETS_BUCKET があれば Cloud Storage
AI 呼び出しlib/ai.ts ＋ lib/config.ts—OPENAI_*（モデル名・画質）で変更可
記名（誰が操作したか）lib/auth.tsブラウザ保持の名前を操作ごとに送る（未設定なら「ゲスト」）ログインは廃止済み。将来は共通ログインに差し替え
裏付け（コードと一致）: lib/store/index.ts は DATABASE_URL 等の有無で Pg*Repository か Json*Repository を選ぶ。lib/images.ts は ASSETS_BUCKET の有無で Cloud Storage かローカル保存を選ぶ。どちらも呼び出し側（画面）は同じインターフェースを使う。
なぜこの形か: 持ち帰って自分のPCでオフラインでも動かせ（JSON＋ローカル画像）、本番では消えない共有先（DB＋クラウド保存）に自動で切り替わる。Cloud Run は使われないと電源が切れ、再起動で中身がまっさらに戻るため、画像とデータは“消えない置き場”に出す必要があった。

7データの関係（ER図）

データは3つのテーブル（用語・議事録・リクエスト）だけ。スキーマは knowledge_ws。関連ワード・タグ・コメント・いいねは“別テーブル”にせず、用語の行の中に配列やJSONとして持つ。テーブル同士をつなぐ外部キーは無く、つながりはアプリ側で「名前の一致」で計算する（疎結合）。

要件定義書の当初案との違い（正直に）: 当初は related_word テーブルと user テーブル（Supabase Auth）に分ける関係モデルを提案していた。実際は、関連語・タグ・いいね・コメントを用語の行に内包し、利用者テーブルは持たない（名前はただの文字列。認証なし）。少人数・内輪利用に合わせて、つなぎはアプリ側の計算（説明文の自動リンク・共通語での2-hop・リクエストの自動解決）で実現している。

8料金の目安と、今できていること

費用が一番動くのはイラストの画質。テキストは3人・少量なら誤差。下の金額は設定モデルから自動計算した「目安（概算）」で、実際の請求は内容で変わる。

10用語ミニ辞典

この資料に出てくる言葉を、比喩を使わず一言で。自分で人に説明するときの“手控え”。

Next.js（App Router）
画面づくりとサーバー処理を1つにまとめて作れるWebアプリの枠組み。このアプリの土台。
Server Action
ブラウザのフォームから直接呼べるサーバー側の処理。登録・編集などの“データを変える操作”はここで行う。
Cloud Run
Googleのクラウドでアプリを動かす場所。使われない間は電源を落として節約し、必要なときに起き上がる。
PostgreSQL
データを保存する共有データベース。このアプリは knowledge_ws という区画（スキーマ）に置く。
Cloud Storage
Googleのクラウドのファイル置き場。消えると困るイラスト画像をここに保存する。
環境変数
プログラムの外から渡す設定値。鍵や接続先をコードに直接書かず、ここで切り替える。
jsonb / text[]
PostgreSQLのデータの型。jsonb＝JSONをそのまま保存／text[]＝文字の配列（複数の語を1つの列に持てる）。
外部キー（FK）
テーブル同士を正式につなぐDBの仕組み。このアプリは使わず、アプリ側で名前を照らし合わせてつなぐ。
WebRTC
ブラウザがサーバーを介さず相手と直接やり取りする通信。リアルタイム文字起こしで使う。
after（Next.js）
画面の返事を返した“後”に裏で処理を続ける仕組み。イラスト生成を待たせないために使う。
