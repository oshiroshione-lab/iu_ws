# UI / UX モダン化リサーチ（2026-06-13）

> 目的：「私たちの辞書」アプリを、きれいでモダンな“プロ級”の見た目・使い心地にするための調べもの。
> 読み方：まず「結論」、次に「詳しく」、最後に「出典」。

---

## 結論（先に要点だけ）

### UI（見た目）でやるべきこと
1. **デザイントークン方式にする。** 色・余白・文字サイズ・角丸を「意味のある名前」（例：`background` / `foreground` / `primary` / `muted` / `border`）で1か所に定義し、画面はその名前だけを使う。これでブレない＆あとから一括で変えられる。Tailwind v4 はこの方式と相性が良い（CSS変数＋`@theme`）。
2. **色は OKLCH 形式で持つ。** 明るさが知覚どおりに揃うので、ライト/ダーク両対応がきれいに作れる。
3. **ダークモードを標準にする。** 2026年は「あれば良い機能」ではなく「あって当たり前」。`.dark` クラスで同じトークンを上書きするだけ。
4. **余白は4/8pxの倍数で統一。** これだけで素人っぽさが消える。
5. **タイポグラフィを主役に。** 見出しは大きく太く、本文は読みやすい行間（leading）。情報の優先順位が一目で分かる。
6. **カード中心のレイアウト＋たっぷりの余白（ミニマル）。** 余計な装飾を削り、いまやるべき1タスクに集中させる。汎用3Dや過剰イラストは古い。

### UX（使い心地）でやるべきこと
1. **AI待ち時間はスケルトン表示。** 空白＋スピナーより体感待ち時間が約40%短い。3〜5行のグレーの帯（長さをバラす）で「これから文章が来る」と予感させる。
2. **AIの状態を見える化。** 「受付→生成中（考え中/検索中/作成中）→完成→失敗」を見分けられるように。失敗は種類を出し、やり直し導線を必ず置く。
3. **5秒超の処理にはキャンセル導線。** 逃げ道がないと“人質”感が出る。
4. **空状態（からっぽ画面）を作り込む。** 「なぜ空か→何のための場所か→次の一手」の3層で、最初の登録へ背中を押す。検索0件も“壊れていない”と安心させて代替案を出す。
5. **マイクロインタラクション。** ホバー/フォーカス/完了時に控えめなアニメ。生きている感じが出る。やり過ぎ厳禁。
6. **アクセシビリティ。** キーボードだけで完結、フォーカスリングを見やすく、状態変化はスクリーンリーダーに通知、コントラスト確保。まず意味のあるHTML、ARIAは必要な所だけ。

## 詳しく

### 1. モダンSaaS UIのトレンド（2026）
- 「機能の多さ」より「ユーザーの自信」を軸に設計する。いま必要なものだけ残す**ミニマル＆余白**が再評価。
- ダッシュボードは**クリーンなデータ可視化＋カードベースのレスポンシブ**が定番。
- ビジュアルは汎用3D・過剰なツルツルCG離れ。**プロダクト固有で人間味のある**方向へ。
- **タイポグラフィが主役級**。太い見出しで読み手の視線を誘導。
- **AIがUXに溶け込む**（スマート入力補完・文脈提案など）。
- **ダークモードは標準装備**に。

### 2. ナレッジ系アプリの設計（Notion / Obsidian / Linear）
- **Obsidian**：双方向リンク（wikiリンク）と関係を見せるグラフ。概念どうしの繋がりを可視化。
- **Notion**：リアルタイム共同編集・ドラッグ&ドロップ・視覚的なシンプルさ。
- 示唆：このアプリの「関連ワード」は、Obsidian的な**用語どうしのリンク/つながり**として育てられる（関連語が既存の用語なら相互リンク、無ければ新規登録への入口）。

### 3. デザインシステムの基礎（色・文字・余白）
- **デザイントークンが土台。** 生の値（`gray-900`）でなく**意味の名前**（`color.text.primary`）で持つと最初からテーマ切替に強い。
- **色**：ブランド色（primary/accent）＋UI色＋ニュートラル。各色に階調とコントラスト基準（アクセシビリティ）。
- **文字**：サイズ・太さ・行間・見出しレベル（H1〜H6）・レスポンシブ規則をトークン化。
- **余白**：4pxまたは8px刻みのスケールで全余白を統一（`xs=4`, `md=16` など）。

### 4. Tailwind v4 + shadcn 流のトークン実装（このアプリで採用しやすい）
- shadcn/ui は**CSS変数の意味ペア**で構成：`bg-background`/`text-foreground`、`border-border`、`ring-ring` など。**ベース＝面の色、`-foreground`＝その上の文字色**。
- 主要トークン：`background`/`foreground`、`card`/`card-foreground`、`popover`、`primary`/`primary-foreground`、`secondary`、`muted`/`muted-foreground`、`accent`、`destructive`、`border`、`input`、`ring`、`chart-1..5`。
- **ダーク**：`.dark` セレクタで同じトークンを上書き。
- **色形式**：OKLCH 推奨（特にダークで色が破綻しにくい）。
  - 例（ライト `:root`）：`--background: oklch(1 0 0)` / `--foreground: oklch(0.145 0 0)` / `--primary: oklch(0.205 0 0)`。
  - 例（`.dark`）：`--background: oklch(0.145 0 0)` / `--foreground: oklch(0.985 0 0)`。
- **角丸**：`--radius`（既定 `0.625rem`）を基準に `sm=0.6× md=0.8× lg=1.0× xl=1.4×…` の派生スケール。

### 5. AI生成中のローディングUX
- **スケルトン > スピナー**：最終UIの形を先に出すと体感待ち時間が短い（テストで約40%短縮）。
- **生成前の遅延（500ms〜2秒）でも画面を空にしない**：スケルトンで埋める。テキスト欄は**3〜5行の幅違いシマー**で自然な行の揺らぎを再現。
- **状態を見分けられるように**：受付（プロンプト受理）→ストリーミング（「考え中／検索中／生成中」のラベル付き）→完成（内容＋出どころ）→失敗（種類別：プロバイダ障害/ポリシー/文脈超過/入力不備）。
- **ストリーミング**：可能なら文字/語/文ごとに逐次表示（“打っている”ように見える）。動きは控えめに、全トークンをアニメさせて読みにくくしない。
- **キャンセル**：長い/間違った生成のために中断導線を用意。状態ラベルは対象コンテンツの近くに常時表示。
- **アクセシビリティ**：キーボードだけで完了できる／状態遷移で焦点順が論理的／状態変化（loading/error/done）をスクリーンリーダーへ通知。

### 6. 空状態・マイクロインタラクション・アクセシビリティ
- **空状態の3層**：①主メッセージ（なぜ空か）②副メッセージ（何のための場所か）③次の一手（どう進むか）。最高到達点は「空状態が楽しい/満足を生む」レベル。
- **検索0件**は高ストレス。すぐに代替案・ガイドを出す。
- **マイクロインタラクション**：進捗の祝福、代替パスの提示、フレンドリーな挿絵で“生きている”感。やり過ぎない。
- **無効状態（disabled）**：押せないと分かりつつ、読めるコントラストを保つバランス。

## このアプリへの当てはめ（実装メモ）
- `app/globals.css` を **OKLCHトークン＋ライト/ダーク** に作り替える（差し替え点は1ファイル）。
- 共通UI部品を `components/ui/` に用意（Button / Card / Input / Badge / Skeleton など）。画面はそれを使うだけにする。
- AI登録の体験：登録ボタン押下 → 詳細ページへ即遷移 → **スケルトン**で「説明・関連ワード・イラスト」の枠を見せる → 出来た所から差し替え（イラストは最後でも可）。失敗は枠ごとに種類別メッセージ＋やり直し。
- 一覧：カードのホバー/フォーカスを洗練。空状態を作り込む。
- ヘッダー：ダークモード切替トグル・現在ユーザー・登録ボタンを整える。
- 既存の「比喩を使わない」「失敗を握りつぶさない」方針は維持（CLAUDE.md / 要件定義 F-02・F-03・F-04）。
